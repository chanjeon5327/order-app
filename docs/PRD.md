#1. 프로젝트 개요

###1.1 프로젝트명
커피 주문 앱

###1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리 할 수 있는 간단한 풀스택 웹 앱

###1.3 개발 범위
-주문하기 화면(메뉴 선택 및 장바구니 기능)
-관리자 화면(재고 관리 및 주문 사태 관리)
-데이터를 생성/조회/수정/삭제 할수 있는 기능

##2. 기술 스택
- 프런트엔드 :html,css, 리액트, 자바스트립트
-백엔드 : node.js,express)
-데이터베이스 : postgreSQL

##3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피메뉴만 있음

##4. 주문하기 화면 UI/UX 명세

###4.1 화면 구조
주문하기 화면은 크게 3개의 섹션으로 구성됩니다:
1. 헤더 섹션
2. 메뉴 아이템 섹션
3. 장바구니 섹션

###4.2 헤더 섹션
**위치**: 화면 최상단
**구성 요소**:
- 좌측: 브랜드 로고/명칭 "COZY" (직사각형 박스 형태)
- 우측: 두 개의 버튼
  - "주문하기" 버튼 (현재 화면 표시용)
  - "관리자" 버튼 (관리자 화면으로 이동)

**기능**:
- "관리자" 버튼 클릭 시 관리자 화면으로 이동

###4.3 메뉴 아이템 섹션
**위치**: 헤더 하단
**레이아웃**: 메뉴 아이템들이 카드 형태로 가로로 배치 (반응형 그리드 레이아웃)

**각 메뉴 카드 구성 요소**:
1. **상품 이미지 영역** (카드 상단)
   - 직사각형 형태의 이미지 표시 영역
   - 이미지가 없을 경우 플레이스홀더 표시

2. **상품명**
   - 예: "아메리카노(ICE)", "아메리카노(HOT)", "카페라떼"

3. **기본 가격**
   - 예: "4,000원", "5,000원" 형식으로 표시
   - 숫자는 천단위 구분 기호(쉼표) 사용

4. **상품 설명**
   - 예: "간단한 설명..." 형태의 텍스트 영역

5. **옵션 선택 영역** (체크박스)
   - 각 옵션은 체크박스 형태로 제공
   - 옵션명과 추가 가격을 함께 표시
     - 예: "샷 추가 (+500원)"
     - 예: "시럽 추가 (+0원)"
   - 사용자가 원하는 옵션을 하나 이상 선택 가능
   - 옵션 선택에 따라 가격이 실시간으로 계산되어야 함 (옵션)

6. **"담기" 버튼**
   - 카드 하단에 배치
   - 클릭 시 선택한 옵션과 함께 장바구니에 추가

**기능**:
- 메뉴 목록 조회 및 표시
- 옵션 선택/해제
- 장바구니에 메뉴 추가
- 동일 메뉴+옵션 조합이 장바구니에 이미 있는 경우 수량 증가 (또는 별도 항목으로 추가)

###4.4 장바구니 섹션
**위치**: 화면 하단 (고정 또는 스크롤 가능 영역)
**레이아웃**: 직사각형 박스 형태로 구분

**구성 요소**:
1. **섹션 제목**: "장바구니"

2. **장바구니 아이템 리스트**
   - 각 아이템은 다음 정보를 포함:
     - 상품명
     - 선택한 옵션 정보 (예: "(샷 추가)")
     - 수량 (예: "X 1", "X 2")
     - 소계 금액 (예: "4,500원", "8,000원")
   - 예시 표시 형식: "아메리카노(ICE) (샷 추가) X 1 - 4,500원"
   - 각 아이템별 삭제 기능 (옵션)

3. **총 금액**
   - "총 금액 12,500원" 형태로 표시
   - 모든 장바구니 아이템의 소계를 합산한 금액
   - 천단위 구분 기호(쉼표) 사용

4. **"주문하기" 버튼**
   - 장바구니 하단에 배치
   - 클릭 시 주문을 완료하는 액션 수행

**기능**:
- 장바구니에 담긴 메뉴 목록 표시
- 각 아이템의 옵션 및 수량 표시
- 실시간 총 금액 계산 및 표시
- 주문 완료 기능

###4.5 사용자 플로우
1. 사용자가 주문하기 화면에 진입
2. 메뉴 카드에서 원하는 메뉴 확인
3. (선택) 메뉴 옵션 선택 (샷 추가, 시럽 추가 등)
4. "담기" 버튼 클릭하여 장바구니에 추가
5. 장바구니 섹션에서 선택한 메뉴 확인
6. 필요시 추가 메뉴를 장바구니에 추가 (2-4 반복)
7. 총 금액 확인
8. "주문하기" 버튼 클릭하여 주문 완료

###4.6 UI/UX 요구사항
- 반응형 디자인: 다양한 화면 크기에 대응
- 직관적인 사용자 인터페이스
- 장바구니 추가 시 시각적 피드백 제공 (옵션)
- 가격 정보가 명확하게 표시되어야 함
- 옵션 선택 시 가격 변동이 명확하게 표시되어야 함 (옵션)

###4.7 데이터 표시 형식
- 가격: 천단위 구분 기호 사용 (예: 4,000원, 12,500원)
- 수량: "X 숫자" 형식 (예: X 1, X 2)
- 옵션: 괄호 안에 표시 (예: (샷 추가))

##5. 관리자 화면 UI/UX 명세

###5.1 화면 구조
관리자 화면은 크게 4개의 섹션으로 구성됩니다:
1. 헤더 섹션
2. 관리자 대시보드 섹션
3. 재고 현황 섹션
4. 주문 현황 섹션

###5.2 헤더 섹션
**위치**: 화면 최상단
**구성 요소**:
- 좌측: 브랜드 로고/명칭 "COZY" (직사각형 박스 형태)
- 우측: 두 개의 버튼
  - "주문하기" 버튼 (주문하기 화면으로 이동)
  - "관리자" 버튼 (현재 화면 표시용, 활성화 상태로 표시)

**기능**:
- "주문하기" 버튼 클릭 시 주문하기 화면으로 이동

###5.3 관리자 대시보드 섹션
**위치**: 헤더 하단
**구성 요소**:
- 섹션 제목: "관리자 대시보드"
- 주문 통계 요약 정보 (한 줄로 표시):
  - 형식: "총 주문 [숫자] / 주문 접수 [숫자] / 제조 중 [숫자] / 제조 완료 [숫자]"
  - 예시: "총 주문 1 / 주문 접수 1 / 제조 중 0 / 제조 완료 0"

**기능**:
- 실시간 주문 상태 통계 표시
- 주문 상태별 개수 집계 및 표시

**주문 상태 종류**:
- 총 주문: 전체 주문 건수
- 주문 접수: 접수된 주문 건수
- 제조 중: 현재 제조 중인 주문 건수
- 제조 완료: 제조가 완료된 주문 건수

###5.4 재고 현황 섹션
**위치**: 대시보드 섹션 하단
**레이아웃**: 각 메뉴 아이템별로 카드 형태로 배치 (가로 배치 또는 그리드 레이아웃)

**각 재고 카드 구성 요소**:
1. **메뉴명**
   - 예: "아메리카노 (ICE)", "아메리카노 (HOT)", "카페라떼"

2. **현재 재고 수량**
   - 형식: "[숫자]개"
   - 예시: "10개"
   - 숫자는 천단위 구분 기호(쉼표) 사용 (필요 시)

3. **재고 조정 버튼**
   - "+" 버튼: 재고 증가 (작은 사각형 버튼)
   - "-" 버튼: 재고 감소 (작은 사각형 버튼)
   - 버튼은 수량 표시 영역 하단에 배치

**기능**:
- 각 메뉴 아이템별 현재 재고 수량 조회 및 표시
- "+" 버튼 클릭 시 해당 메뉴의 재고 수량 증가
- "-" 버튼 클릭 시 해당 메뉴의 재고 수량 감소
- 재고 수량 실시간 업데이트

**제약 사항** (옵션):
- 재고 수량이 0 미만으로 내려가지 않도록 제한 (옵션)
- 재고 수량 변경 시 확인 절차 (옵션)

###5.5 주문 현황 섹션
**위치**: 재고 현황 섹션 하단
**레이아웃**: 주문 목록을 리스트 형태로 표시

**구성 요소**:
1. **섹션 제목**: "주문 현황"

2. **주문 목록**
   - 각 주문 항목은 다음 정보를 포함:
     - **주문 일시**: "월 일일 시:분" 형식 (예: "7월 31일 13:00")
     - **주문 내용**: 상품명과 수량 (예: "아메리카노(ICE) x 1")
     - **주문 금액**: "가격원" 형식 (예: "4,000원")
       - 천단위 구분 기호(쉼표) 사용
     - **상태 버튼**: 주문 상태에 따른 버튼 (우측 끝에 배치)
       - 예: "주문 접수" 버튼 (현재 상태 표시 또는 상태 변경 액션)

**기능**:
- 주문 목록 조회 및 표시
- 최신 주문이 상단에 표시 (시간순 정렬)
- 각 주문의 상태 확인
- 상태 버튼 클릭 시 주문 상태 변경
  - 상태 변경 흐름: 주문 접수 → 제조 중 → 제조 완료 (또는 단계별 변경)

**주문 상태 변경 플로우** (옵션):
1. 새 주문 생성 시: "주문 접수" 상태
2. "제조 중" 버튼 클릭: 제조 중으로 상태 변경
3. "제조 완료" 버튼 클릭: 제조 완료로 상태 변경

**표시 형식 예시**:
- 일시: 7월 31일 13:00
- 내용: 아메리카노(ICE) x 1
- 금액: 4,000원
- 버튼: [주문 접수]

###5.6 사용자 플로우
1. 관리자가 관리자 화면에 진입
2. 대시보드에서 전체 주문 통계 확인
3. 재고 현황에서 각 메뉴의 재고 수량 확인
4. 필요시 재고 수량 조정 (+/- 버튼 사용)
5. 주문 현황에서 새로 들어온 주문 확인
6. 주문 상태 버튼 클릭하여 주문 상태 변경 (주문 접수 → 제조 중 → 제조 완료)

###5.7 UI/UX 요구사항
- 반응형 디자인: 다양한 화면 크기에 대응
- 직관적인 관리자 인터페이스
- 재고 수량 변경 시 시각적 피드백 제공
- 주문 상태 변경 시 명확한 피드백 제공
- 실시간 데이터 업데이트 (옵션: 자동 새로고침 또는 수동 새로고침)
- 상태별 색상 구분 (옵션: 예 - 접수: 파란색, 제조 중: 주황색, 완료: 초록색)

###5.8 데이터 표시 형식
- 재고 수량: "[숫자]개" 형식 (예: 10개, 1,000개)
- 주문 일시: "월 일일 시:분" 형식 (예: 7월 31일 13:00)
- 주문 수량: "x 숫자" 형식 (예: x 1, x 2)
- 가격: 천단위 구분 기호 사용 (예: 4,000원, 12,500원)

##6. 백엔드 개발 명세

###6.1 데이터 모델

####6.1.1 Menus (메뉴)
메뉴 정보를 저장하는 테이블입니다.

**필드 구성**:
- `id` (Primary Key) - 메뉴 고유 ID
- `name` (String) - 커피 이름 (예: "아메리카노(ICE)", "카페라떼")
- `description` (String) - 메뉴 설명
- `price` (Integer) - 기본 가격 (원 단위)
- `image` (String) - 이미지 URL 또는 경로
- `stock` (Integer) - 재고 수량
- `created_at` (Timestamp) - 생성 일시
- `updated_at` (Timestamp) - 수정 일시

**제약 사항**:
- `name`, `price`, `stock`는 필수 항목
- `price`는 0 이상의 정수
- `stock`는 0 이상의 정수

####6.1.2 Options (옵션)
메뉴 옵션 정보를 저장하는 테이블입니다.

**필드 구성**:
- `id` (Primary Key) - 옵션 고유 ID
- `name` (String) - 옵션 이름 (예: "샷 추가", "시럽 추가")
- `price` (Integer) - 옵션 추가 가격 (원 단위, 0 이상)
- `menu_id` (Foreign Key) - 연결된 메뉴 ID (Menus 테이블 참조)
- `created_at` (Timestamp) - 생성 일시
- `updated_at` (Timestamp) - 수정 일시

**제약 사항**:
- `name`, `price`, `menu_id`는 필수 항목
- `price`는 0 이상의 정수
- 하나의 메뉴는 여러 옵션을 가질 수 있음 (1:N 관계)

####6.1.3 Orders (주문)
주문 정보를 저장하는 테이블입니다.

**필드 구성**:
- `id` (Primary Key) - 주문 고유 ID
- `order_date` (Timestamp) - 주문 일시
- `menu_id` (Foreign Key) - 주문한 메뉴 ID (Menus 테이블 참조)
- `quantity` (Integer) - 주문 수량
- `total_price` (Integer) - 주문 총 금액 (메뉴 가격 + 옵션 가격) * 수량
- `status` (String) - 주문 상태 ("received": 주문 접수, "making": 제조 중, "completed": 제조 완료)
- `created_at` (Timestamp) - 생성 일시
- `updated_at` (Timestamp) - 수정 일시

**주문 내용 상세 정보** (JSON 또는 별도 테이블):
- 선택한 옵션 정보 (Options 테이블의 ID 배열 또는 JSON)
- 옵션별 추가 가격 정보

**제약 사항**:
- `order_date`, `menu_id`, `quantity`, `total_price`, `status`는 필수 항목
- `quantity`는 1 이상의 정수
- `total_price`는 0 이상의 정수
- `status`는 "received", "making", "completed" 중 하나

**옵션 정보 저장 방식** (선택):
- 방법 1: `selected_options` (JSON 필드)에 옵션 ID 배열 저장
- 방법 2: 별도 `OrderOptions` 테이블 생성 (Orders와 Options의 다대다 관계)

###6.2 데이터 스키마를 위한 사용자 흐름

####6.2.1 메뉴 조회 및 표시
1. **프런트엔드**: 주문하기 화면 진입 시 메뉴 목록 조회 요청
2. **백엔드**: Menus 테이블에서 모든 메뉴 정보 조회
   - 메뉴 ID, 이름, 설명, 가격, 이미지 정보 반환
   - 재고 수량(`stock`)은 관리자 화면에서만 사용하므로 일반 조회 시 제외 가능
3. **프런트엔드**: 받은 메뉴 정보를 화면에 카드 형태로 표시
4. **프런트엔드**: 각 메뉴의 옵션 정보 조회 (Options 테이블에서 `menu_id`로 필터링)
5. **백엔드**: 해당 메뉴의 옵션 목록 반환
6. **프런트엔드**: 옵션을 체크박스 형태로 표시

####6.2.2 장바구니 관리
1. **프런트엔드**: 사용자가 메뉴 선택 및 옵션 선택
2. **프런트엔드**: 선택 정보를 클라이언트 측 장바구니 상태에 저장 (데이터베이스 저장 전)
3. **프런트엔드**: 장바구니에 선택한 메뉴, 수량, 옵션 정보 표시

####6.2.3 주문 생성
1. **프런트엔드**: 사용자가 "주문하기" 버튼 클릭
2. **프런트엔드**: 장바구니의 모든 아이템 정보를 주문 요청으로 전송
3. **백엔드**: Orders 테이블에 주문 정보 저장
   - 각 장바구니 아이템을 개별 주문 레코드로 저장
   - `order_date`: 현재 시간
   - `status`: "received" (주문 접수)
   - 선택한 옵션 정보도 함께 저장
4. **백엔드**: 주문된 메뉴의 재고 수량 차감
   - Menus 테이블에서 해당 메뉴의 `stock` 값을 주문 수량만큼 감소
   - 재고가 부족한 경우 에러 반환
5. **백엔드**: 주문 생성 완료 응답 반환
6. **프런트엔드**: 주문 완료 메시지 표시 및 장바구니 초기화

####6.2.4 주문 현황 조회
1. **프런트엔드**: 관리자 화면 진입 시 주문 목록 조회 요청
2. **백엔드**: Orders 테이블에서 모든 주문 정보 조회
   - 최신 주문이 먼저 오도록 정렬 (`order_date` 내림차순)
   - 주문 상태별 필터링 가능
3. **백엔드**: 각 주문의 메뉴 정보 조회 (Menus 테이블 조인)
4. **백엔드**: 각 주문의 옵션 정보 조회 (Options 테이블 조인 또는 JSON 파싱)
5. **프런트엔드**: 주문 목록을 관리자 화면의 "주문 현황" 섹션에 표시

####6.2.5 주문 상태 변경
1. **프런트엔드**: 관리자가 주문 상태 버튼 클릭 ("제조 시작" 또는 "제조 완료")
2. **프런트엔드**: 주문 ID와 새로운 상태를 백엔드로 전송
3. **백엔드**: Orders 테이블에서 해당 주문의 `status` 필드 업데이트
   - 상태 변경 흐름: "received" → "making" → "completed"
4. **백엔드**: 업데이트 완료 응답 반환
5. **프런트엔드**: 주문 목록 새로고침 또는 해당 주문만 업데이트

####6.2.6 재고 관리
1. **프런트엔드**: 관리자 화면에서 Menus 테이블의 재고 정보 조회
2. **백엔드**: Menus 테이블에서 모든 메뉴의 재고 수량(`stock`) 반환
3. **프런트엔드**: 재고 현황 섹션에 표시
4. **프런트엔드**: 관리자가 재고 조정 버튼(+/-) 클릭
5. **백엔드**: Menus 테이블에서 해당 메뉴의 `stock` 값 업데이트
6. **백엔드**: 업데이트 완료 응답 반환
7. **프런트엔드**: 재고 정보 새로고침

###6.3 API 설계

####6.3.1 메뉴 관련 API

**GET /api/menus**
- **설명**: 모든 메뉴 목록 조회
- **요청**: 없음
- **응답**:
  ```json
  [
    {
      "id": 1,
      "name": "아메리카노(ICE)",
      "description": "간단한 설명...",
      "price": 4000,
      "image": "https://...",
      "stock": 10
    },
    ...
  ]
  ```
- **용도**: 주문하기 화면에서 메뉴 목록 표시

**GET /api/menus/:id**
- **설명**: 특정 메뉴 정보 조회
- **요청**: URL 파라미터로 메뉴 ID 전달
- **응답**: 메뉴 상세 정보
- **용도**: 메뉴 상세 정보 조회

**GET /api/menus/:id/options**
- **설명**: 특정 메뉴의 옵션 목록 조회
- **요청**: URL 파라미터로 메뉴 ID 전달
- **응답**:
  ```json
  [
    {
      "id": 1,
      "name": "샷 추가",
      "price": 500,
      "menu_id": 1
    },
    ...
  ]
  ```
- **용도**: 메뉴 선택 시 옵션 목록 표시

**PUT /api/menus/:id/stock**
- **설명**: 메뉴 재고 수량 업데이트
- **요청**:
  ```json
  {
    "stock": 15
  }
  ```
- **응답**: 업데이트된 메뉴 정보
- **용도**: 관리자 화면에서 재고 수량 조정

####6.3.2 주문 관련 API

**POST /api/orders**
- **설명**: 새 주문 생성
- **요청**:
  ```json
  {
    "items": [
      {
        "menu_id": 1,
        "quantity": 2,
        "selected_options": [1, 2],
        "total_price": 9000
      },
      ...
    ]
  }
  ```
- **응답**:
  ```json
  {
    "success": true,
    "order_ids": ["order_123", "order_124"],
    "message": "주문이 완료되었습니다."
  }
  ```
- **용도**: 장바구니에서 주문하기 버튼 클릭 시
- **비즈니스 로직**:
  1. 각 주문 아이템을 Orders 테이블에 저장
  2. 주문된 메뉴의 재고 수량 차감 (Menus 테이블 업데이트)
  3. 재고 부족 시 에러 반환

**GET /api/orders**
- **설명**: 모든 주문 목록 조회
- **요청**: 쿼리 파라미터 (선택)
  - `status`: 주문 상태 필터 ("received", "making", "completed")
- **응답**:
  ```json
  [
    {
      "id": "order_123",
      "order_date": "2024-01-15T10:30:00Z",
      "menu_id": 1,
      "menu_name": "아메리카노(ICE)",
      "quantity": 2,
      "total_price": 9000,
      "status": "received",
      "options": [
        {
          "id": 1,
          "name": "샷 추가",
          "price": 500
        }
      ]
    },
    ...
  ]
  ```
- **용도**: 관리자 화면의 주문 현황 표시

**GET /api/orders/:id**
- **설명**: 특정 주문 정보 조회
- **요청**: URL 파라미터로 주문 ID 전달
- **응답**: 주문 상세 정보 (메뉴, 옵션, 금액 등 포함)
- **용도**: 주문 상세 정보 확인

**PUT /api/orders/:id/status**
- **설명**: 주문 상태 변경
- **요청**:
  ```json
  {
    "status": "making"
  }
  ```
- **응답**: 업데이트된 주문 정보
- **용도**: 관리자 화면에서 주문 상태 변경 ("제조 시작", "제조 완료" 버튼)
- **상태 변경 규칙**:
  - "received" → "making" (제조 시작)
  - "making" → "completed" (제조 완료)
  - "completed" 상태는 변경 불가

###6.4 에러 처리

**공통 에러 응답 형식**:
```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "에러 메시지"
  }
}
```

**주요 에러 코드**:
- `INSUFFICIENT_STOCK`: 재고 부족
- `INVALID_ORDER_ID`: 잘못된 주문 ID
- `INVALID_STATUS`: 잘못된 주문 상태
- `NOT_FOUND`: 리소스를 찾을 수 없음
- `VALIDATION_ERROR`: 입력 데이터 검증 실패

###6.5 데이터베이스 스키마 예시

**Menus 테이블**:
```sql
CREATE TABLE menus (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  description TEXT,
  price INTEGER NOT NULL CHECK (price >= 0),
  image VARCHAR(500),
  stock INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Options 테이블**:
```sql
CREATE TABLE options (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  price INTEGER NOT NULL DEFAULT 0 CHECK (price >= 0),
  menu_id INTEGER NOT NULL REFERENCES menus(id) ON DELETE CASCADE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Orders 테이블**:
```sql
CREATE TABLE orders (
  id VARCHAR(50) PRIMARY KEY,
  order_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  menu_id INTEGER NOT NULL REFERENCES menus(id),
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  total_price INTEGER NOT NULL CHECK (total_price >= 0),
  status VARCHAR(20) NOT NULL CHECK (status IN ('received', 'making', 'completed')),
  selected_options JSONB,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_order_date ON orders(order_date DESC);
```

###6.6 API 구현 우선순위

1. **1단계 (필수)**:
   - GET /api/menus (메뉴 목록 조회)
   - GET /api/menus/:id/options (옵션 목록 조회)
   - POST /api/orders (주문 생성)
   - GET /api/orders (주문 목록 조회)

2. **2단계 (중요)**:
   - PUT /api/orders/:id/status (주문 상태 변경)
   - PUT /api/menus/:id/stock (재고 수정)
   - GET /api/inventory (재고 현황 조회)

3. **3단계 (보조)**:
   - GET /api/orders/stats (통계 정보)
   - GET /api/orders/:id (주문 상세 조회)
   - GET /api/menus/:id (메뉴 상세 조회)
